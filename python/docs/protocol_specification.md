# edLoRa Protocol Specification

## Overview
edLoRa is a lightweight, cross-language (C++ & Python) binary protocol designed specifically for high-power rocketry telemetry and command data over LoRa modules. It prioritizes low overhead, reliability, and cross-platform compatibility (Linux, ESP32, Python).

## Packet Structure
Every packet generated by `edLoRa` strictly adheres to a binary format consisting of a fixed-size header, a variable-length payload, and a fixed-size footer (CRC16).

| Segment | Field | Size (Bytes) | Type | Description |
| :--- | :--- | :--- | :--- | :--- |
| **HEADER** | `SYNC_BYTE` | 1 | `uint8_t` | Constant `0xED`. Used for stream framing and alignment. |
| | `version` | 1 | `uint8_t` | Protocol Version. Currently strictly `0x02`. |
| | `flags` | 1 | `uint8_t` | Bitmask for packet properties (ACK, Encrypted, Fragmented). |
| | `sender_id` | 1 | `uint8_t` | ID of the transmitting node. |
| | `receiver_id` | 1 | `uint8_t` | Targeted node ID. `0xFF` used for Broadcasts. |
| | `msg_type` | 1 | `uint8_t` | Identifier for payload parsing (e.g., Altimeter, Command). |
| | `seq_num` | 1 | `uint8_t` | Rolling sequence integer (0-255) to detect dropped packets. |
| | `timestamp` | 4 | `uint32_t` | Embedded millisecond clock natively injected at tx-time. (Little Endian) |
| | `payload_len` | 1 | `uint8_t` | Length of the dynamic payload segment (0-240 bytes). |
| **PAYLOAD** | `payload` | 0 - 240 | `uint8_t[]`| Arbitrary binary data. Format relies strictly on `msg_type`. |
| **FOOTER** | `crc16` | 2 | `uint16_t` | CRC-16 CCITT calculated over Header + Payload. (Little Endian) |

- **Total Header Size:** 12 bytes
- **Total Footer Size:** 2 bytes
- **Max Packet Size:** 254 bytes (12 Header + 240 Payload + 2 Footer)

## Message Types
`edLoRa` natively utilizes a `MsgType` Enum mapped statically across both Python and C++.

| MsgType | Hex (`uint8_t`) | Standard Usage |
| :--- | :--- | :--- |
| `HEARTBEAT` | `0x00` | Periodic pings to verify connection integrity. |
| `GPS` | `0x01` | Lat, Lon, Alt datagrams. |
| `ALTIMETER` | `0x02` | Barometric Altitude, Vertical Velocity datagrams. |
| `IMU` | `0x03` | Accelerometer, Gyroscope structs. |
| `COMMAND` | `0x04` | Uplink instructions (`DEPLOY CHUTE`). |
| `SYS_STATE` | `0x05` | Battery level, Temperature, active phase. |
| `ORIENTATION`| `0x06` | Pre-calculated Quaternions or Euler calculations. |
| `EVENT` | `0x07` | Critical one-shot states (e.g. `APOGEE_DETECTED`). |
| `ACK` | `0xFD` | Automated Acknowledgement packet (Payload = original `seq_num`). |
| `ERROR_MSG` | `0xFE` | Device Faults. |
| `CUSTOM` | `0xFF` | Generic arbitrary type. |

## ACK Command System
To implement safe multi-node commanding environments (like Ground Station firing sequences), `edLoRa` includes an ACK response system natively built onto the `Packet` class (`.create_ack()`).

1. A targeted `MsgType::COMMAND` packet arrives at Node A from Node B with `seq_num = 42`.
2. Node A automatically generates an ACK and reflects it directly back to Node B via `create_ack(my_id, timestamp)`.
3. The generated packet intrinsically holds `msg_type = MsgType::ACK`, automatically swaps the target/sender IDs, and embeds the parent message's pure sequence number (`42`) as the sole byte of its payload array.

## Stream Recovery & Framing
When intercepting streams from hardware (e.g., UART byte streams from ESP32 or SX1276 radios), half-packets and dropped buffers are highly common.
The protocol is engineered to recover gracefully by searching sequentially for the protocol's `SYNC_BYTE` (`0xED`). If a buffer misaligns or parses garbage data, the CRC-16 routine automatically rejects the frame ensuring your `unpack()` operations only ever return `true` on statistically perfect bit-frames.
